          Pros/Cons, lưu ý và cải thiện đề xuất

          Pros
          • Không ảnh hưởng flow payout class/instructor đang vận hành ổn định.
          • Giảm rủi ro rollout vì Wallet chỉ thêm lớp ghi nhận + event payout.
          • Dễ triển khai mobile history/detail do taxonomy giao dịch rõ theo stream.
          Cons
          • Tồn tại song song 2 lane payout: event qua wallet, class qua legacy.
          • Báo cáo payout tổng cần hợp nhất từ 2 nguồn.
          • Ops cần training rõ để tránh xử lý nhầm policy.
          Lưu ý/Cải thiện
          • Bắt buộc gắn nhãn stream ở payout request và transfer record.
          • Thêm cảnh báo UI khi user thao tác payout class (đi legacy flow).
          • Tạo dashboard reconciliation tách rõ Event Wallet và Instructor Legacy.

              Context Note (BA/PO/PM)

              Cho Design
              Thiết kế phải ưu tiên clarity cho end-user: tổng quan số dư, lịch sử tra cứu, và context giao dịch.
              Cho Dev
              Triển khai theo stream Event/Course nhất quán ở filter, detail, và rule payout.
              Cho Client
              Mockup thể hiện rõ phạm vi phase hiện tại: Event payout trong Wallet, Course/Class payout ở Teaching Space.
              Chọn giao dịch để xem ghi chú thực thi cho Design/Dev/Client.
              Pros
              • Phù hợp SOW dài hạn (ALL users) ngay từ kiến trúc dữ liệu.
              • Giảm rủi ro rollout nhờ triển khai theo phase thay vì big-bang.
              • UX end-to-end vẫn đầy đủ: entry → dashboard → history → detail → payout.
              Cons
              • Trong ngắn hạn vẫn phải quản lý song song Event Wallet và Course legacy.
              • Cần truyền thông rõ cho user về phạm vi thao tác mỗi phase.
              • Báo cáo tài chính cần map chuẩn giữa lane mới và lane legacy.
              Lưu ý / Đề xuất
              • Phase 1-2 (gộp): hoàn thiện Wallet foundation cho cả Event/Course.
              • Phase 3A/B/C: triển khai trọn gói Payout System theo đúng 2.1-2.9 trong SOW.
